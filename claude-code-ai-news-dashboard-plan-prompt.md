## 역할

너는 시니어 풀스택 엔지니어이자 데이터 파이프라인 설계자다. 지금은 **플랜 모드**다. 코드를 작성하지 말고, 먼저 아래 요구사항을 충족하는 구현 계획(아키텍처, 기술 선택과 트레이드오프, 파일 구조, 단계별 작업, 비용 추정, 리스크)을 제안하라. 내가 계획을 승인한 뒤에만 구현을 시작한다.

## 만들 것

**완전 자동화된 글로벌 AI 뉴스 팔로업 대시보드 웹사이트.** 매일 자동으로 전 세계 주요 AI 소식을 수집·정리·요약해 보여주는 정적(혹은 서버리스) 사이트다. 사람이 손대지 않아도 매일 갱신되어야 한다.

> 프로젝트 루트는 `C:\workspace\ai-daily-news`다(Windows 환경). 모든 파일·디렉터리·스크립트 경로는 이 루트 기준으로 설계하고, OS 의존 부분(경로 구분자, 셸 명령, 줄바꿈 등)은 Windows 호환성을 고려하라. 단, GitHub Actions 등 자동화 실행 환경은 Linux 러너일 수 있으니 로컬(Windows)과 CI(Linux) 양쪽에서 동작하도록 설계하라.

## 핵심 요구사항 (기능)

1. **자동 수집**: 아래 소스에서 RSS/API로 신규 글을 정기적으로 가져온다.
2. **중복 제거**: 같은 사건을 여러 매체가 다룰 때 묶거나 중복을 제거한다.
3. **카테고리 분류**: 최소 — 모델/연구, 비즈니스/투자, 정책/규제, 제품/툴, 하드웨어/인프라.
4. **전문 읽기 → 인사이트 도출 → 요약**: 각 기사의 **본문 전체를 가져와 LLM이 읽고**, 단순 발췌가 아니라 핵심 인사이트(무엇이 새로운지, 왜 중요한지, 비즈니스/기술적 함의)를 도출한 **한국어 요약 + 인사이트 2~3문장**을 생성한다.
   - **원문 전문은 저장·노출하지 않는다.** 본문은 LLM 입력용으로만 쓰고, 출력물은 LLM이 새로 작성한 한국어 요약·인사이트뿐이다.
   - 각 카드에는 원제·매체·발행시각과 함께 **원문 링크**를 두어, 클릭하면 **실제 원본 사이트로 이동**하게 한다(전문 복제 대체가 아니라 원문 유도).
   - 이 요약·인사이트 생성은 **Claude Haiku** 모델로 수행한다(매일 대량 처리 비용 최소화).
5. **랭킹/하이라이트**: 최신성과 중요도(언급 빈도, 소스 신뢰도 등)를 반영해 상단에 "오늘의 핵심 5건"을 노출한다.
6. **대시보드 UI**: 반응형. 카테고리/소스/날짜 필터, 키워드 검색, 트렌딩 키워드, "마지막 업데이트 시각" 표시.
7. **아카이브**: 날짜별로 과거 기록을 조회할 수 있게 한다.

## 데이터 소스 (초기 후보 — 가용한 RSS/API를 검증해 확정하라)

- **뉴스 매체**: TechCrunch(AI), The Verge(AI), VentureBeat(AI), MIT Technology Review, Wired(AI), Ars Technica
- **연구**: arXiv (cs.AI, cs.LG, cs.CL 카테고리 API)
- **기업/랩 공식 블로그**: OpenAI, Anthropic, Google DeepMind, Meta AI, Hugging Face
- **커뮤니티 신호**: Hacker News(AI 키워드 필터링)

> 각 소스의 RSS 엔드포인트 존재 여부, robots.txt 및 이용약관, 요청 빈도 제한을 점검하고, 막힌 소스는 대안을 제시하라. RSS에 본문 전체가 없으면 원문 페이지에서 본문을 가져와 **LLM 입력용으로만** 사용한다(저장·노출 금지, 출력은 LLM이 새로 쓴 요약뿐). 무단 대량 스크래핑은 피하고 robots.txt를 존중하라.

## 자동화 & 배포 요구사항

- **스케줄링**: 매일 최소 1회(가능하면 하루 2~3회) 자동 실행. **GitHub Actions cron**을 우선 검토하되 대안(Cloudflare Workers Cron, Vercel Cron)도 비교하라.
- **무인 운영**: 수집 → 처리 → 빌드 → 배포가 사람 개입 없이 끝까지 자동으로 이어져야 한다.
- **호스팅**: 비용 최소화를 위해 정적 사이트 + 무료 호스팅(GitHub Pages / Cloudflare Pages / Vercel)을 우선 고려하라.
- **비밀키 관리**: API 키는 환경변수/Secrets로만 다루고 절대 코드·저장소에 노출하지 않는다.
- **데이터 저장**: 수집 결과의 저장 방식(JSON 파일 커밋 vs 경량 DB vs KV 스토어)을 트레이드오프와 함께 제안하라.

## 기술 스택 가이드 (제안하되 최종 결정은 근거와 함께 네가 하라)

- 프론트엔드: Next.js(정적 export) / Astro / 순수 정적 SSG 중 비교.
- 수집·처리 파이프라인: Python 또는 Node.js.
- 비용·유지보수성·확장성을 기준으로 추천 스택을 1안으로 확정하고 1개의 대안을 함께 제시하라.

## 제약사항

- **저비용/무료 우선**, 운영 부담 최소화.
- 소스의 저작권·이용약관 준수: 전문 복제 금지, 요약 + 원문 링크 원칙.
- 한 사람이 유지보수 가능한 단순한 구조.
- API 호출 비용(요약 LLM)을 추정하라. 요약·인사이트 생성은 **Claude Haiku**로 처리하는 전제이며, 비용 절감 방안(신규 기사만 처리, 처리 결과 캐싱, 본문 길이 상한 설정, 중복 묶음 후 대표 기사만 요약 등)을 포함하라.

## 플랜 모드 산출물 (이 형식으로 답하라)

1. **아키텍처 개요** — 데이터 흐름 다이어그램(텍스트로) 포함
2. **기술 스택 결정** — 선택지별 트레이드오프와 최종 추천(1안 + 대안)
3. **소스 검증 결과** — 각 소스의 RSS/API 가용성과 대안
4. **파일/디렉터리 구조**
5. **단계별 구현 계획(Phase 1~N)** — 각 단계의 산출물과 검증 방법
6. **자동화 파이프라인 설계** — 스케줄, 트리거, 실패 시 처리
7. **예상 비용** — 월 운영비(호스팅 + Haiku API) 추정. 일일 기사 수 × 본문 토큰 × Haiku 단가 기준으로 계산 근거를 제시하라.
8. **리스크 및 미해결 질문** — 내게 확인이 필요한 의사결정 사항

계획을 제시한 뒤, 진행 전에 내 확인이 필요한 핵심 결정 사항을 질문으로 정리해 달라.
